32  The Ch4_Data_Choreography space
33  (DistributedData/BoxPattern and subclasses) is meant to provide a
34  reduced complexity, highly maintainable alternative to the Copier and
35  LevelData infrastructure. If you need high performance or need
36  some of the fancier aspects of Copier, you should use the
37  standard LevelData/BoxLayoutData/LayoutData/Copier
38  infrastructure.
39 
40  Periodic boundary conditions are not supported.
41 
42  The applications for which this is intended do not need the optimizations
43  around which LevelData is built and do not need periodic boundary conditions.
44 
45  All communication is two phase.
46  All communication is done on the host.
47  We aggregate all communications between two processors the same way LevelData does.
48 
49  DistributedData holds data over a union of rectangles.
50  Each rectangle is a Box in a DisjointBoxLayout.
51  There are two communication patterns associated with this data.
52  1. copyTo: where one DistributedData writes its data to another
53  over the intersection of their DisjointBoxLayouts. Ghost data is *not* overwritten.
54  This is slightly different behavior than standard Chombo3 LevelData.
55  2. exchange: ghost cell information where ghost data from one grid
56  is filled with valid data from a neighboring grid.
57 
58  DistributedData --- holds data and manages communcation.
59  CommunicationMeta --- deals with the communication of meta data information
60  (also holds the string buffers used to call MPI)
61  BoxPattern --- Manages all the on-processor meta data
62  (who is talking to whom via what boxes)
63  box_interaction_t the meta data associated with two boxes communciating
64  proc_interaction_t the meta data associated with two procs communicating
65 */
